This would be a domain like example.com, with no subdomains, including the www subdomain. However, that doesn't mean that all businesses (or I'd argue most) are able to require a subdomain, even www, for all traffic. You simply can't stop users from typing in example.com no matter how much you market www.example.com. They have sound points as to why you want to avoid them but there are a few tricks to make this work as reliably and securely as possible. After discussing with their support team on several issues, here's the final solution. Just to be very clear and up front, I'm not an SSL or DNS expert, but these solutions came from people who are. I try to elaborate on the "why" as much as possible but if you have an extra insight that I don't explain enough or don't explain properly, please let me know so that I can update the points here :)

Setting up your SSL Endpoint

If you aren't using SSL, just skip this section. Since you should be using SSL if you submit any sort of sensitive data to your app over the interwebs (passwords included) then you should probably do this. First, do the steps to Acquire your SSL Certificate and set up the SSL Endpoint on your app if you haven't already. One thing that surprised me is that just using the standard heroku certs:add command caused there to be a missing chain certificate and therefore sometimes still throwing a browser warning about the certificate. I know basically nothing about this but I can tell you what I did the first time that was wrong. I downloaded the certificate I purchased from Go Daddy that included two files: gd_bundle.crt and mydomain.com.crt. When I ran heroku certs:add I did it as heroku certs:add my domain.com.crt server.key like the doc says. I guess what I was missing was the gd_bundle.crt to have the proper SSL trust chain. Apparently, "The gd_bundle.crt is an intermediate certificate bundle that is used to complete the chain of trust on your certificate installation". I'm not sure why it's not in their primary documentation or the primary way to add certificates, but Heroku created an SSL Doctor tool to set this up properly for you. So install the plugin by running heroku plugins:install https://github.com/heroku/heroku-ssl-doctor.git and then run heroku certs:add path/to/certs/* where the path/to/certs/ directory has your gd_bundle.crt (or whatever your SSL certificate authority gave you), the yourdomain.com.crt and your server.key file. If you already set up your SSL endpoint and just need to fix it, use heroku certs:update path/to/certs/* instead. Use a DNS provider that allows you to map your naked domain to a dynamic hostname

The first issue to solve is that the typical setup of a naked domain on Heroku requires you to set your A record to three static IP addresses. The problem with this is that there is no failover ability if there is an issue with these IP addresses. If they go down or are under some sort of DOS attack, then your application goes down. When you map a subdomain to your Heroku application's hostname using a CNAME record (for instance, mapping www.example.com to example.herokuapp.com) then Heroku has the ability to do whatever is needed behind the scenes to make the hostname available to you. The other issue is that these three IP addresses go into their older Bamboo stack and most new applications run on the Cedar stack, so there's some extra latency in your application when using these IP addresses. Most DNS providers don't have the ability to use a CNAME-style mapping of your naked domain to a dynamic hostname like this. There are, however, at least three that do: DNSimple, DNS Made Easy, and Amazon Web Services' Route 53. I've had my eye on DNSimple for a while, so I opted to go with them. Their interface seems to imply that you need to transfer registration of your domain to them but that's not really the case. I kept the domain registered at Go Daddy but changed to their name servers, however, I will probably also transfer registration to them as well when it's time to renew. Waiting to transfer registrars also makes this possible without any downtime for the domain. The record you want to add a DNSimple is an ALIAS record rather than an A record. They act the same way but with ALIAS you are mapping to a dynamic hostname rather than an IP. So trash the three A records for your domain pointing to Heroku's IP addresses and add one ALIAS record for your naked domain pointing to your-ssl-endpoint.herokussl.com if you are using SSL or yourapp.herokuapp.com if you aren't using an SSL certificate. I'm not sure if all of those providers call that record an ALIAS record, so my apologies if they call it something else.

Set up the www subdomain

Just like Heroku says, create a CNAME record that maps the www subdomain to your-ssl-endpoint.herokussl.com if you are using SSL or yourapp.herokuapp.com if you aren't using an SSL certificate. You may also want to consider taking Heroku's advice and redirect all naked domain traffic to use www as well. I'm assuming there is some sort of internal latency and/or extra possibility of failure on your DNS providers side when it comes to the ALIAS record since it has to internally look up an IP to return for the naked domain.
